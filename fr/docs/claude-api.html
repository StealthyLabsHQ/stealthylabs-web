1 Vue d'ensemble

L'API Claude est une API HTTP RESTful accessible à https://api.anthropic.com qui donne un accès programmatique aux modèles Claude. Tu envoies des messages, Claude répond. C'est le principe de base. Tout le reste – streaming, tool use, vision, caching – s'appuie sur cette primitive.

L'endpoint principal est le Messages API (POST /v1/messages). Il gère les requêtes simples, les conversations multi-tours, l'analyse d'images, l'appel de fonctions, et bien plus.

Ce qu'on peut builder

  • Chatbots et assistants conversationnels
  • Outils de génération et revue de code
  • Pipelines d'analyse et de résumé de documents
  • Agents autonomes qui appellent des outils et APIs externes
  • Applications multimodales avec compréhension d'images
  • Extraction de données structurées depuis du texte non structuré
  • Génération de contenu à grande échelle via le Batch API

APIs disponibles

APIStatutObjectif
Messages APIGAAPI conversationnelle principale
Message Batches APIGATraitement asynchrone à -50% de coût
Token Counting APIGACompter les tokens avant envoi pour maîtriser les coûts
Models APIGALister les modèles disponibles et leurs détails
Files APIBetaUploader des fichiers une fois, les référencer par ID
Où accéder à Claude : Directement via l'API Anthropic (accès prioritaire aux nouvelles fonctionnalités), ou via AWS Bedrock, Google Cloud Vertex AI et Microsoft Azure (intégrés à la facturation et l'IAM cloud).

2 Démarrage

1
Créer un compte Anthropic Console
Aller sur console.anthropic.com et s'inscrire. Vérifier son email et optionnellement ajouter une carte bancaire pour accéder aux limites de taux supérieures.
2
Obtenir sa clé API
Dans la Console, aller dans API Keys et créer une nouvelle clé. La copier immédiatement – elle n'est affichée qu'une fois. La stocker de façon sécurisée (variable d'environnement, gestionnaire de secrets).
3
Installer le SDK ou utiliser cURL
Des SDKs officiels existent pour Python, TypeScript/Node.js, Java, Go, C#, Ruby et PHP. Pour des tests rapides, cURL fonctionne directement.
4
Faire son premier appel
Envoyer un message et recevoir une réponse. Voir les exemples ci-dessous.

3 Modèles

Choisir le bon modèle pour son cas d'usage. Le compromis est toujours entre capacité, vitesse et coût.

Modèle ID API Idéal pour Contexte Prix (entrée/sortie par MTok)
Claude Opus 4.6 claude-opus-4-6 Tâches complexes, agents, raisonnement poussé, code 1M tokens $5 / $25
Claude Sonnet 4.6 claude-sonnet-4-6 Meilleur équilibre – charges de production, tâches générales 1M tokens $3 / $15
Claude Haiku 4.5 claude-haiku-4-5-20251001 Volume élevé, faible latence, tâches sensibles au coût 200k tokens $1 / $5

Comment choisir

  • Commencer avec Sonnet 4.6 pour la plupart des cas de production – excellente qualité à coût raisonnable.
  • Utiliser Opus 4.6 quand on a besoin d'une intelligence maximale : workflows agentiques complexes, raisonnement nuancé, code difficile.
  • Utiliser Haiku 4.5 pour la classification, extraction, résumé à grande échelle où la vitesse et le coût priment.
  • Les trois modèles supportent l'extended thinking, la vision (images) et le tool use.
Date de coupure : Opus 4.6 – mai 2025. Sonnet 4.6 – août 2025. Haiku 4.5 – fév. 2025. Claude n'a pas d'accès internet en temps réel sauf si tu lui fournis un outil web_search.

4 Authentification

Chaque requête à l'API nécessite deux headers :

HeaderValeurRequis
x-api-keyTa clé API (sk-ant-...)Oui
anthropic-version2023-06-01Oui
content-typeapplication/jsonOui (pour POST)

Avec un SDK, l'authentification est gérée automatiquement depuis la variable d'environnement ANTHROPIC_API_KEY.

Sécuriser sa clé

  • Ne jamais committer de clés API en versioning. Ajouter .env au .gitignore.
  • Utiliser des variables d'environnement ou un gestionnaire de secrets (AWS Secrets Manager, Vault, etc.) en production.
  • Faire tourner les clés immédiatement en cas de suspicion de fuite.
  • Utiliser le champ metadata.user_id pour tracer l'usage par utilisateur sans envoyer de données personnelles.
Côté serveur uniquement : Ne jamais exposer la clé API dans du JavaScript côté client. Tous les appels à l'API Anthropic doivent passer par le backend.

5 Messages API

POST https://api.anthropic.com/v1/messages est l'endpoint principal. Il accepte une conversation et retourne la réponse de Claude.

Paramètres requis

ParamètreTypeDescription
modelstringID du modèle à utiliser (ex. claude-sonnet-4-6)
max_tokensintegerNombre maximum de tokens que Claude peut générer. Ne garantit pas ce nombre – Claude peut s'arrêter avant.
messagesarrayTableau d'objets {"role": "user"|"assistant", "content": ...}.

Paramètres optionnels

ParamètreTypeDescription
systemstring ou arrayPrompt système – instructions globales pour le comportement de Claude.
temperaturefloat 0–1Aléatoire. Défaut 1.0. Plus bas = plus focalisé/déterministe.
streambooleanActiver le streaming par server-sent events.
stop_sequencesstring[]Arrêter la génération quand Claude produit l'une de ces chaînes.
toolsarrayDéfinitions des outils que Claude peut appeler.
tool_choiceobjectContrôler la sélection d'outil : auto, any, none, ou un nom d'outil spécifique.
top_pfloatNucleus sampling. Utiliser temperature ou top_p, pas les deux.
metadataobject{"user_id": "..."} – identifiant opaque pour le suivi des abus.

// Contenu simple en string

// Tableau de content blocks (requis pour images/documents)

// Conversation multi-tours
[
  {"role": "user",      "content": "Combien font 2+2 ?"},
  {"role": "assistant", "content": "4"},
  {"role": "user",      "content": "Et 4+4 ?"}

6 Prompts Système

Le paramètre system définit le contexte global du comportement de Claude. Il s'exécute avant tout message utilisateur. Utilise-le pour définir persona, contraintes, format de sortie et connaissances domaine. 

client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="Tu es un ingénieur backend senior. Réponds uniquement en TypeScript. "
           "Ajoute toujours des commentaires JSDoc. N'utilise jamais le type 'any'.",
    messages=[{"role": "user", "content": "Écris une fonction pour debouncer des appels."}]
)

Quoi mettre dans un prompt système

  • Persona et ton – "Tu es un agent de support client amical pour StealthyLabs."
  • Format de sortie – "Réponds toujours en JSON. N'ajoute jamais de markdown."
  • Contraintes – "Réponds uniquement aux questions sur notre produit. Décline poliment toute autre demande."
  • Connaissances domaine – Coller des documents de référence, FAQs, specs produit.
  • Instructions chain-of-thought – "Avant de répondre, réfléchis au problème étape par étape."
Conseil fenêtre de contexte : Pour les prompts système volumineux et statiques qui ne changent pas entre les appels, utiliser le prompt caching (voir section 12) pour réduire drastiquement les coûts.

7 Conversations Multi-tours

Le Messages API est sans état. Pour maintenir le contexte entre les tours, il faut envoyer l'historique complet de la conversation à chaque requête. L'application est responsable du stockage et de la gestion du tableau de messages. 

import anthropic

client = anthropic.Anthropic()
conversation = []

def chat(user_message: str) -> str:
    conversation.append({"role": "user", "content": user_message})

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=conversation,
    )

    assistant_reply = response.content[0].text
    conversation.append({"role": "assistant", "content": assistant_reply})
    return assistant_reply

print(chat("Quelle est la capitale de la France ?"))
print(chat("Et quelle est sa population ?"))  # Claude se souvient de Paris

Gérer les limites de la fenêtre de contexte

  • Sonnet 4.6 et Opus 4.6 supportent jusqu'à 1 million de tokens de contexte.
  • Pour de très longues conversations, implémenter une fenêtre glissante, une stratégie de résumé ou de récupération.
  • Utiliser le Token Counting API (POST /v1/messages/count_tokens) pour estimer les coûts avant envoi.
# Compter les tokens avant d'envoyer
token_count = client.messages.count_tokens(
    model="claude-sonnet-4-6",
    messages=conversation,
)
print(f"Cette requête utilisera ~{token_count.input_tokens} tokens d'entrée")

8 Streaming

Par défaut, l'API attend la réponse complète avant d'envoyer quoi que ce soit. Avec stream: true, Claude envoie les tokens au fur et à mesure de leur génération via des Server-Sent Events (SSE). Les UIs semblent beaucoup plus réactives. 

# Python – streaming avec SDK
with client.messages.stream(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Écris un haïku sur l'océan."}],
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)
// TypeScript – streaming avec SDK
const stream = client.messages.stream({
  model: "claude-sonnet-4-6",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Écris un haïku sur l'océan." }],
});

for await (const chunk of stream) {
  if (chunk.type === "content_block_delta" && chunk.delta.type === "text_delta") {
    process.stdout.write(chunk.delta.text);
  }
}
const finalMessage = await stream.finalMessage();
console.log("\nUsage:", finalMessage.usage);

Types d'événements SSE

ÉvénementQuand il se déclenche
message_startDébut de la réponse – contient l'ID du modèle et l'usage initial
content_block_startUn nouveau bloc de contenu commence (text, tool_use, thinking)
content_block_deltaDelta de texte incrémental (les tokens réels)
content_block_stopUn bloc de contenu se termine
message_deltaMet à jour stop_reason et les compteurs de tokens finaux
message_stopStream terminé
Utiliser le streaming pour les interfaces de chat et partout où l'utilisateur attend. Pour le traitement par lots ou les jobs en arrière-plan, le mode non-streaming est plus simple à gérer.

9 Vision (Images)

Tous les modèles Claude actuels peuvent comprendre les images. Inclure des blocs de contenu image dans les messages, aux côtés du texte. Formats supportés : JPEG, PNG, GIF, WebP.

Trois façons de fournir des images

A
Encodage Base64 – Encoder l'image et l'inclure en ligne. Bien pour les requêtes ponctuelles.
B
Référence URL – Fournir une URL publiquement accessible. Claude télécharge l'image. Rapide à implémenter.
C
Files API – Uploader une fois, réutiliser par file_id. Idéal pour les conversations multi-tours et l'usage répété pour éviter de renvoyer des payloads volumineux.
# Python – image par URL
message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[{
        "role": "user",
        "content": [
            {
                "type": "image",
                "source": {"type": "url", "url": "https://example.com/image.jpg"},
            },
            {"type": "text", "text": "Qu'est-ce que tu vois dans cette image ?"}
        ],
    }],
)

Limites et bonnes pratiques

LimiteValeur

  • Redimensionner les grandes images à ≤ 1,15 mégapixels avant upload – aucun gain de qualité au-delà.
  • Claude ne peut pas identifier des personnes, générer/modifier des images ni traiter du contenu inapproprié.

10 Tool Use (Appel de fonctions)

Les tools permettent à Claude d'appeler des fonctions définies par l'utilisateur. On décrit ce que l'outil fait et ses paramètres ; Claude décide quand l'utiliser, construit des arguments valides, et on exécute la fonction pour retourner le résultat. Cela permet de créer des agents qui interagissent avec des APIs, bases de données et services externes.

La boucle tool use

1
Envoyer une requête avec tools définis.
2
Claude répond avec stop_reason: "tool_use" et un bloc tool_use contenant le nom de l'outil et ses arguments.
3
Ton code exécute l'outil avec les arguments fournis.
4
Renvoyer le résultat comme message tool_result.
5
Claude lit le résultat et continue (appel d'un autre outil éventuellement, ou réponse textuelle finale).
import anthropic, json

client = anthropic.Anthropic()

tools = [{
    "name": "get_weather",
    "description": "Obtenir la météo actuelle d'une ville.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "Nom de la ville, ex. Paris"}
        },
        "required": ["city"]
    }
}]

messages = [{"role": "user", "content": "Quel temps fait-il à Tokyo ?"}]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=messages,
)

while response.stop_reason == "tool_use":
    tool_use_block = next(b for b in response.content if b.type == "tool_use")

    # Exécuter l'outil (ta vraie logique ici)
    if tool_use_block.name == "get_weather":
        result = {"temperature": "18°C", "condition": "Nuageux"}

    messages.append({"role": "assistant", "content": response.content})
    messages.append({
        "role": "user",
        "content": [{
            "type": "tool_result",
            "tool_use_id": tool_use_block.id,
            "content": json.dumps(result),
        }]
    })

    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )

print(response.content[0].text)

11 Extended Thinking

L'extended thinking laisse Claude "réfléchir à voix haute" avant de répondre. Il traite les problèmes complexes étape par étape avant de s'engager dans une réponse, améliorant la précision sur les tâches de raisonnement difficile et de maths. Supporté par tous les modèles Claude 4. 

response = client.messages.create(
    model="claude-opus-4-6",
    max_tokens=16000,
    thinking={
        "type": "enabled",
        "budget_tokens": 10000,  # Combien de tokens Claude peut utiliser pour réfléchir
    },
    messages=[{
        "role": "user",
        "content": "Prouve qu'il existe une infinité de nombres premiers."
    }],
)

for block in response.content:
    if block.type == "thinking":
        print("Réflexion de Claude :\n", block.thinking)
    elif block.type == "text":
        print("Réponse finale :\n", block.text)
  • budget_tokens contrôle la limite haute des tokens de réflexion. Plus de budget = potentiellement meilleur raisonnement, coût plus élevé.
  • Les blocs de thinking apparaissent avant la réponse textuelle dans response.content.
  • À utiliser pour : preuves mathématiques, raisonnement multi-étapes, décisions d'architecture code complexes, problèmes logiques difficiles.

12 Prompt Caching

Le prompt caching permet de marquer des parties du prompt comme cachables. Sur les requêtes suivantes qui incluent le même préfixe mis en cache, les tokens cachés sont lus au lieu d'être retraités – donnant une réduction de coût d'~90% et une réduction de latence d'~85% sur ces tokens. 

# Cacher un grand prompt système
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "Tu es un assistant IA spécialisé dans l'analyse de documents juridiques. "
                    + tres_long_document_reference,  # Des milliers de tokens
            "cache_control": {"type": "ephemeral"},
        }
    ],
    messages=[{"role": "user", "content": "Résume la clause 7."}],
)

# response.usage.cache_creation_input_tokens  → tokens écrits en cache (premier appel)
# response.usage.cache_read_input_tokens      → tokens lus depuis le cache (appels suivants)

Quand utiliser le caching

  • Grands prompts système statiques qui se répètent sur de nombreuses requêtes
  • Documents de référence, catalogues produits ou FAQs inclus dans chaque appel
  • Longs historiques de conversation en sessions multi-tours
  • Exemples few-shot qui ne changent pas
Note sur le prix : Les écritures en cache coûtent 25% de plus que les tokens d'entrée normaux. Les lectures coûtent environ 10% des tokens d'entrée normaux. L'investissement est rentabilisé après 2 lectures.

13 Sorties Structurées

Il existe deux façons principales d'obtenir des données structurées de façon fiable depuis Claude :

Méthode 1 – Prompt + validation

Dire à Claude de répondre en JSON dans le prompt système, puis parser et valider avec une bibliothèque (Pydantic pour Python, Zod pour TypeScript).

from pydantic import BaseModel
import json

class CritiqueFil m(BaseModel):
    titre: str
    note: float
    resume: str
    points_forts: list[str]
    points_faibles: list[str]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="Réponds toujours en JSON valide uniquement. Pas de markdown, pas d'explication.",
    messages=[{"role": "user", "content": "Critique Inception (2010)."}],
)

data = json.loads(response.content[0].text)
critique = CritiqueFilm(**data)  # Pydantic valide le schéma

Méthode 2 – Tool use pour une structure garantie

Définir un outil avec le schéma JSON exact. Forcer Claude à l'appeler avec tool_choice. Claude retournera toujours du JSON valide correspondant au schéma.

tools = [{
    "name": "extraire_donnees",
    "description": "Extraire des données structurées du texte.",
    "input_schema": {
        "type": "object",
        "properties": {
            "titre":  {"type": "string"},
            "note":   {"type": "number", "minimum": 0, "maximum": 10},
            "resume": {"type": "string"},
        },
        "required": ["titre", "note", "resume"]
    }
}]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    tool_choice={"type": "tool", "name": "extraire_donnees"},
    messages=[{"role": "user", "content": "Critique Inception (2010)."}],
)

tool_block = next(b for b in response.content if b.type == "tool_use")
data = tool_block.input  # Déjà un dict Python correspondant au schéma

14 Traitement par Lots

Le Message Batches API traite de grands volumes de requêtes de façon asynchrone à -50% de coût. Idéal pour traiter des milliers de documents, des évaluations, ou tout workload qui n'a pas besoin de réponses en temps réel. 

# Créer un lot pouvant aller jusqu'à 100 000 requêtes
batch = client.messages.batches.create(
    requests=[
        {
            "custom_id": "avis-1",
            "params": {
                "model": "claude-haiku-4-5-20251001",
                "max_tokens": 256,
                "messages": [{"role": "user", "content": "Classfie le sentiment : 'Super produit !'"}],
            },
        },
        {
            "custom_id": "avis-2",
            "params": {
                "model": "claude-haiku-4-5-20251001",
                "max_tokens": 256,
                "messages": [{"role": "user", "content": "Classifie le sentiment : 'Expérience terrible.'"}],
            },
        },
    ]
)

# Attendre et récupérer les résultats
import time
while True:
    status = client.messages.batches.retrieve(batch.id)
    if status.processing_status == "ended":
        break
    time.sleep(60)

for result in client.messages.batches.results(batch.id):
    print(result.custom_id, result.result.message.content[0].text)

15 Limites de Taux & Erreurs

StatusCode d'erreurSignificationAction
400invalid_request_errorRequête malformée, champ requis manquantCorriger la requête
401authentication_errorClé API invalide ou manquanteVérifier la clé
403permission_errorLa clé manque de permissionsVérifier les scopes
429rate_limit_errorTrop de requêtesBackoff exponentiel
500api_errorErreur serveur AnthropicRéessayer après délai
529overloaded_errorAPI surchargéeRéessayer après délai
 
import time, anthropic
from anthropic import RateLimitError, APIStatusError

client = anthropic.Anthropic()

def creer_avec_retry(max_retries=5, **kwargs):
    for attempt in range(max_retries):
        try:
            return client.messages.create(**kwargs)
        except RateLimitError:
            if attempt == max_retries - 1:
                raise
            wait = 2 ** attempt  # 1s, 2s, 4s, 8s, 16s
            print(f"Rate limited. Retry dans {wait}s...")
            time.sleep(wait)
        except APIStatusError as e:
            if e.status_code in (500, 529) and attempt < max_retries - 1:
                time.sleep(2 ** attempt)
            else:
                raise

16 Prompt Engineering

De bons prompts déterminent 80% de la qualité de sortie. Voici les techniques à haut rendement spécifiques à Claude.

Être spécifique sur ce qu'on veut

FaibleFort

"Écris du code.""Écris une fonction TypeScript avec JSDoc. Utilise async/await. Inclus la gestion d'erreurs. Retourne un résultat typé, pas any."

Chain-of-thought

system = "Avant de répondre à toute question, écris ton raisonnement étape par étape dans des balises <thinking>. Puis fournis ta réponse finale."

Exemples few-shot

messages = [{
    "role": "user",
    "content": """Classifie le sentiment de chaque avis comme POSITIF, NÉGATIF ou NEUTRE.

17 Sécurité

  • Clés API côté serveur uniquement. Ne jamais les exposer dans du JavaScript navigateur, des apps mobiles ou des dépôts publics.
  • Valider et assainir les entrées utilisateur avant de les passer dans les prompts pour mitiger les attaques par injection de prompt.
  • Implémenter son propre rate limiting sur les endpoints pour prévenir les abus des utilisateurs de l'application.
  • Ne jamais faire confiance à la sortie de Claude comme vérité absolue pour les décisions sensibles à la sécurité (authentification, autorisation, transactions financières).
  • Faire tourner les clés API régulièrement et immédiatement en cas de suspicion de fuite.
  • Logger les requêtes et réponses pour les pistes d'audit, particulièrement dans les secteurs réglementés.
Injection de prompt : Du contenu malveillant dans le texte fourni par l'utilisateur peut tenter d'écraser les instructions du prompt système. Valider les entrées, utiliser des champs séparés pour les données utilisateur et les instructions, et appliquer une validation des sorties.
